# Skill Harvest — frontload-audit must RESOLVE the paid-resource gap into a budget policy, not defer it

- **target skill:** `/loopgen` (SKILL.md Phase 3 frontload-audit + `primitives/runner-contract.md`; archetype bodies `templates/bodies/goal-body.md` and `frontier-body.md`).
- **observed gap:** loopgen's frontload-audit lists "claude-cli runs cost real API $ — treat as a soft budget" under `gaps_flagged` and STOPS there. For an **unattended** loop a "soft budget" with no operative policy is not a deferrable gap — it is a runtime deadlock: the first paid step either (a) fires an interactive `AskUserQuestion` (the loop idles until a human wakes), or (b) surprise-spends with no ceiling. A gap flagged "soft budget, keep iterations small" reads as resolved to the composer but is actually an unwritten control-flow branch the loop hits on night 1. The composed PROMPT.md had a Judgment-default sentence ("prefer local or a 1-task probe, log it") and an escalate clause ("paid-API without budget cap") but NO autonomous cap mechanism — so the only path the loop had at the first frontier dollar was to escalate-and-idle, which is exactly the unattended failure mode.
- **evidence iteration:** mdtools hybrid-pareto `/goal` loop, 2026-05-29 overnight kickoff. Confirmed on disk: STATE.md `iteration: 0`, `loop/runs/init-frontier/` empty, `init-local/` holding only a 0-byte `unix.txt` + one in-flight T1 log — the loop stopped mid-Bootstrap at step 5 (the paid `init-frontier` sweep) and fired an `AskUserQuestion`. Compounding latent gaps the audit also missed and a later adversarial pass surfaced: (1) the cap was drafted in **USD** but `bench/harness.py:1189-1195` measures **only tokens** and the repo has **no token→USD price table** (`grep` for price/usd/per_token/Mtok across `bench/ loop/` = nothing) — every dollar gate was uncomputable; (2) the **bootstrap inventory sweep** (24 tasks × 3 modes × N3 ≈ 6× the seed-family anchor) is the FIRST paid step and CREATES the cells, so a cap sized for the seed family funds nothing and the deferral machinery had no cell to key on; (3) the human-settable cap was placed in **loop-rewritten STATE.md** — a lost-update race against the loop's end-of-iteration full-file write.
- **proposed rule:** frontload-audit must classify any flagged resource as **METERED-vs-UNMETERED** and **REVERSIBLE-vs-IRREVERSIBLE**. For any "iteration spends real money / consumes an irreversible external resource," the audit may NOT leave it as a bare `gaps_flagged` line — it MUST **resolve it into an operative budget/quota policy** in PROMPT.md before emit, with five mandatory properties: (a) the cap is denominated in the unit the harness actually MEASURES (here: tokens, since no $ price table exists) — verify measurability by grepping the harness for the unit; (b) the human-authored ceiling lives on a **READ-only** surface (PROMPT.md / env), never in loop-rewritten state; (c) over-cap work is **deferred (logged + resumable)**, never AskUserQuestion and never idle; (d) the spend ledger is **write-ahead** so a mid-iteration crash fails CLOSED; (e) the **one-time bootstrap/instantiation cost is costed separately** from per-unit work and surfaced so a cap number is never silently insufficient.
- **why it generalizes:** every runner in the family (`/goal`, ralph, gnhf, cocc) re-invokes the same prompt unattended; "spend real money / burn an irreversible quota" is a property of the *task's resources*, not this repo. Any loop with a metered paid tier (API $, cloud compute, rate-limited external API, irreversible publish) hits the identical fork at its first paid step. The frontier and goal archetypes are the exposed ones (both can carry a paid verifier/tier); story/greenfield are exposed whenever they call a paid model. The USD-vs-measured-unit trap and the bootstrap-cost-vs-per-unit-cost trap recur wherever the cap unit differs from what the harness logs.
- **suggested patch wording (Phase 3 frontload-audit checklist, NOT yet applied):** add a gate — "For each flagged resource: is it METERED and IRREVERSIBLE (spends money / burns a non-replenishable quota)? If yes, it CANNOT remain a `gaps_flagged` line — RESOLVE it into a `## Budget policy` (or `## Quota policy`) section with: cap in the harness-MEASURED unit (grep the harness to confirm the unit is observable; if the human wants another unit, pin an explicit conversion table); read-only cap surface; defer-don't-block + never-idle; write-ahead ledger (crash fails closed); per-atom re-check (no mid-flight kill assumption); and a separately-costed bootstrap/instantiation line. A loop that can spend unattended without all six is not emittable." Cross-reference from `goal-body.md` / `frontier-body.md` Judgment-default so the body's "costs real $" sentence points at the resolved section instead of standing alone.
- **accidental-encouragement risk:** an over-broad "always write a full budget-policy section" rule could push operators to bolt a heavyweight token-ledger onto loops whose paid tier is trivially bounded (a single fixed-cost call, a free-only loop) — ceremony where a one-line cap suffices. Mitigation: the full policy is required ONLY when the loop can spend **repeatedly / unboundedly while unattended** (a per-iteration or per-cell paid reach). For a single bounded paid action, a one-line authorized-or-defer rule is enough; for free-only loops, no budget section is emitted at all. Tie the requirement to "unattended + repeatable paid reach," not to "touches money at all."
